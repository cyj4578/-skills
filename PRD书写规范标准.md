# 产品需求文档（PRD）规范标准

> **版本**：v1.0  
> **创建日期**：2026-04-16  
> **适用场景**：所有产品需求文档的编写遵循本规范  
> **维护责任人**：产品团队

---

## 一、文档目的

本文档定义了叫号小程序项目及后续所有产品需求文档（PRD）的书写规范。所有团队成员在编写、评审和执行需求时，必须遵循本规范所定义的结构和标准。

---

## 二、PRD 文档结构

每份 PRD 必须包含以下核心章节，顺序固定，不可缺失。

### 2.1 文档封面信息

```markdown
| 字段 | 内容 |
|------|------|
| 文档标题 | [功能/模块名称] 需求文档 |
| 文档编号 | PRD-[年份]-[序号] |
| 产品版本 | v[X.X] |
| 作者 | [姓名] |
| 创建日期 | YYYY-MM-DD |
| 最后更新 | YYYY-MM-DD |
| 状态 | 草稿 / 待评审 / 已评审 / 已批准 / 已上线 |
| 关联需求 | [相关 Issue/PR/设计稿链接] |
```

### 2.2 文档修订历史

| 版本 | 日期 | 作者 | 变更说明 |
|------|------|------|----------|
| v0.1 | YYYY-MM-DD | 张三 | 初始版本 |
| v1.0 | YYYY-MM-DD | 张三 | 完成评审，批准执行 |

## 三、核心章节说明

### 3.1 问题陈述（Problem Statement）

**目的**：阐明为什么要做这个需求。

**要求**：
- 描述用户面临的具体问题或痛点
- 说明受影响的用户群体
- 量化问题的严重程度和影响范围
- 用数据或案例支撑（如有）

**模板**：
> 目前 [用户/场景] 面临 [具体问题]，导致 [负面影响]。根据 [数据来源]，约有 [X%] 的用户遇到此问题，平均每次 [损失/耗时] [Y]。如果不解决，将影响 [业务目标]。

---

### 3.2 目标（Goals）

**目的**：明确需求上线后要达成的可衡量成果。

**要求**：
- 3-5 个具体、可量化的目标
- 每个目标包含指标名称、目标值、衡量时间
- 与业务目标对齐

**模板**：
| 目标编号 | 目标描述 | 衡量指标 | 目标值 | 衡量时间 |
|----------|----------|----------|--------|----------|
| G-01 | [描述] | [指标] | [值] | 上线后 X 天 |
| G-02 | ... | ... | ... | ... |

---

### 3.3 非目标（Non-Goals）

**目的**：明确排除的范围，防止需求蔓延。

**要求**：
- 3-5 项明确不在本期范围内的内容
- 简要说明排除原因
- 列出未来可能考虑的方向

**模板**：
| 非目标 | 排除原因 |
|--------|----------|
| [功能点] | [原因，如：技术复杂度高、优先级低、需独立评审] |
| ... | ... |

---

### 3.4 用户故事（User Stories）

**目的**：从用户视角描述功能价值。

**要求**：
- 使用标准格式：`作为 <用户角色>，我想要 <功能>，以便 <获得价值>`
- 按用户角色分组
- 优先级从高到低排序
- 每个故事对应验收标准

**模板**：
```
用户角色：[角色名称]

【P0 - 必须】
- US-01：作为 [用户]，我希望 [功能]，以便 [价值]
  - 验收标准：
    1. [条件] 时，[期望结果]
    2. ...

【P1 - 重要】
- US-02：...

【P2 - 期望】
- US-03：...
```

---

### 3.5 需求详细说明（Requirements）

**目的**：将用户故事拆解为可执行的技术需求。

**5.5.1 需求分类**

| 优先级 | 标识 | 定义 | 交付要求 |
|--------|------|------|----------|
| 必须有 | P0 | 核心功能，缺失则需求不成立 | 必须在 MVP 中交付 |
| 应该有 | P1 | 重要功能，显著提升用户体验 | 计划在本期交付 |
| 可以有 | P2 | 优化功能，有则锦上添花 | 根据资源情况决定 |
| 未来考虑 | P3 | 长期规划，当前不实现 | 记录但不开发 |

**5.5.2 功能需求（FR）**

| 需求ID | 需求描述 | 所属用户故事 | 优先级 | 验收标准 |
|--------|----------|--------------|--------|----------|
| FR-01 | [描述] | US-01 | P0 | [标准1]<br>[标准2] |
| FR-02 | ... | US-02 | P1 | ... |

**5.5.3 非功能需求（NFR）**

| 类型 | 需求描述 | 衡量标准 |
|------|----------|----------|
| 性能 | 页面加载时间 | 首屏 < 2s |
| 性能 | 接口响应时间 | < 500ms |
| 兼容性 | 微信版本 | 支持 7.0+ |
| 安全性 | 数据传输 | HTTPS 加密 |
| 可用性 | 系统可用性 | 99.9% |
| 监控 | 错误率 | < 0.1% |

---

### 3.6 成功指标（Success Metrics）

**目的**：定义如何衡量需求是否成功。

**要求**：
- 区分领先指标（快速变化）和滞后指标（长期变化）
- 设定具体的目标值和衡量周期
- 说明数据来源和计算方法

**模板**：
| 指标类型 | 指标名称 | 目标值 | 当前值 | 提升幅度 | 衡量周期 | 数据来源 |
|----------|----------|--------|--------|----------|----------|----------|
| 领先指标 | [如：功能使用率] | [值] | [值] | +X% | 上线后 7 天 | 埋点数据 |
| 滞后指标 | [如：用户满意度] | [值] | [值] | +X% | 上线后 30 天 | 问卷/NPS |

---

### 3.7 开放问题（Open Questions）

**目的**：记录待解决但影响开发的问题。

**要求**：
- 标注问题所属领域（产品/设计/研发/测试/法务）
- 明确负责人和期望答复时间
- 定期跟踪直至关闭

**模板**：
| 问题 | 领域 | 负责人 | 期望答复 | 状态 | 备注 |
|------|------|--------|----------|------|------|
| [问题描述] | 产品 | @张三 | YYYY-MM-DD | 待回复 | ... |

---

### 3.8 时间计划（Timeline & Milestones）

**目的**：明确关键时间节点和交付物。

**模板**：
| 里程碑 | 计划日期 | 实际日期 | 交付物 | 状态 |
|--------|----------|----------|--------|------|
| PRD 评审 | YYYY-MM-DD | - | 完成评审的 PRD | 待开始 |
| 设计交付 | YYYY-MM-DD | - | 设计稿 + 设计说明 | - |
| 开发完成 | YYYY-MM-DD | - | 代码合入主分支 | - |
| 测试完成 | YYYY-MM-DD | - | 测试报告 | - |
| 灰度发布 | YYYY-MM-DD | - | X% 用户可用 | - |
| 全量上线 | YYYY-MM-DD | - | 100% 用户可用 | - |

### 3.10 附录（Appendix）

根据需要添加：

- **术语表**：文档中使用的专业术语解释

## 四、书写规范

### 4.1 语言与格式

| 规范项 | 要求 |
|--------|------|
| 文档语言 | 中文为主，术语保持一致 |
| 标题层级 | H1（文档标题）→ H2（章节）→ H3（子节）→ H4（条款） |
| 列表层级 | 最多 3 级，避免过度嵌套 |
| 表格 | 必须包含表头，字段对齐 |
| 链接 | 使用相对路径或完整 URL，便于追溯 |

### 4.2 需求命名规范

| 类型 | 命名格式 | 示例 |
|------|----------|------|
| 用户故事 | US-XX | US-01, US-02 |
| 功能需求 | FR-XX | FR-01, FR-02 |
| 非功能需求 | NFR-XX | NFR-01 |
| 目标 | G-XX | G-01, G-02 |
| 问题 | Q-XX | Q-01 |

### 4.3 状态定义

| 状态 | 定义 |
|------|------|
| 草稿 | 编写中，尚未完成 |
| 待评审 | 已完成，等待评审 |
| 评审中 | 评审进行中 |
| 已评审 | 评审通过，有修改意见待处理 |
| 已批准 | 所有意见已处理，文档冻结 |
| 开发中 | 已开始开发 |
| 已上线 | 功能已全量发布 |
| 已归档 | 需求已废弃或合并 |

## 五、评审流程

### 5.1 评审节点

```
[PRD 编写] → [产品负责人初审] → [评审会] → [修改完善] → [批准发布]
     ↓              ↓              ↓            ↓             ↓
   状态：草稿    状态：待评审   状态：评审中  状态：已评审   状态：已批准
```

### 5.2 评审 Checklist

在提交评审前，作者需确认：

- [ ] 文档结构完整，无缺失章节
- [ ] 问题陈述清晰，有数据支撑
- [ ] 目标可衡量，指标明确
- [ ] 非目标明确，防止范围蔓延
- [ ] 用户故事覆盖所有核心场景
- [ ] 需求优先级已与干系人确认
- [ ] 验收标准具体可测试
- [ ] 成功指标可量化、可追踪
- [ ] 开放问题已标注负责人
- [ ] 时间计划合理可行
- [ ] 风险已识别并有应对措施
- [ ] 术语一致，无歧义
- [ ] 链接有效，可追溯

## 六、示例：叫号小程序 - 排队取号功能 PRD

### 6.1 文档封面

```markdown
| 字段 | 内容 |
|------|------|
| 文档标题 | 叫号小程序 - 排队取号功能需求文档 |
| 文档编号 | PRD-2026-001 |
| 产品版本 | v1.0 |
| 作者 | 产品经理 |
| 创建日期 | 2026-04-16 |
| 最后更新 | 2026-04-16 |
| 状态 | 已批准 |
```

### 6.2 问题陈述

> 顾客到店后，需要在前台人工登记排队，效率低下且容易出错。根据调研，约有 **35%** 的顾客因排队等待时间不确定而放弃等候，导致客流流失。此外，前台人员手动登记增加了工作负担，平均每笔登记耗时 **45 秒**。

### 6.3 目标

| 目标编号 | 目标描述 | 衡量指标 | 目标值 | 衡量时间 |
|----------|----------|----------|--------|----------|
| G-01 | 提高取号效率 | 顾客自助取号耗时 | ≤ 15s | 上线后 7 天 |
| G-02 | 减少客流流失 | 放弃等候率 | 降低 20% | 上线后 30 天 |
| G-03 | 减轻前台负担 | 前台登记笔数 | 减少 60% | 上线后 30 天 |

### 6.4 非目标

| 非目标 | 排除原因 |
|--------|----------|
| 预约功能 | 需独立评审，本期仅实现现场取号 |
| 多店铺管理 | 属于商家端功能，放至二期 |
| 叫号语音播报 | 硬件依赖复杂，后期规划 |
| 历史排队数据分析 | 数据基础设施尚未就绪 |

### 6.5 用户故事

```
用户角色：顾客

【P0 - 必须】
- US-01：作为顾客，我希望扫码快速取号，以便无需排队等候
  - 验收标准：
    1. 扫描门店二维码后，5秒内显示取号页面
    2. 点击「立即取号」后，显示当前排队号码和预计等待时间
    3. 取号成功后，收到微信服务通知

【P1 - 重要】
- US-02：作为顾客，我希望查看实时排队进度，以便合理安排时间
  - 验收标准：
    1. 页面显示当前叫号进度（当前叫到 XX 号）
    2. 每当有新号被叫时，页面自动刷新进度

【P2 - 期望】
- US-03：作为顾客，我希望过号后重新排队，以便灵活处理突发情况
  - 验收标准：
    1. 过号超过 3 人后，可选择「重新取号」
    2. 系统保留原号码最后 3 位数字作为历史记录
```

### 6.6 功能需求

| 需求ID | 需求描述 | 所属用户故事 | 优先级 | 验收标准 |
|--------|----------|--------------|--------|----------|
| FR-01 | 扫码入口 | US-01 | P0 | 微信扫描二维码后，自动打开小程序指定页面 |
| FR-02 | 门店信息展示 | US-01 | P0 | 显示门店名称、营业状态、当前等候人数 |
| FR-03 | 取号接口调用 | US-01 | P0 | 调用后端 API 生成排队号，返回号码 + 位置 + 预计等待 |
| FR-04 | 服务通知推送 | US-01 | P0 | 取号成功推送通知，包含号码和查询链接 |
| FR-05 | 实时进度查询 | US-02 | P1 | WebSocket 长连接推送叫号进度 |
| FR-06 | 过号重排机制 | US-03 | P2 | 过号用户可重新取号，保留历史号码片段 |

### 6.7 非功能需求

| 类型 | 需求描述 | 衡量标准 |
|------|----------|----------|
| 性能 | 小程序首屏加载 | ≤ 2s（4G 网络） |
| 性能 | 取号接口响应 | ≤ 300ms |
| 兼容性 | 微信版本 | 支持 7.0 及以上 |
| 稳定性 | 接口可用性 | ≥ 99.9% |
| 监控 | 错误日志 | 接入微信小程序监控 |

### 6.8 成功指标

| 指标类型 | 指标名称 | 目标值 | 当前值 | 衡量周期 | 数据来源 |
|----------|----------|--------|--------|----------|----------|
| 领先指标 | 自助取号使用率 | ≥ 70% | 0% | 上线后 14 天 | 埋点数据 |
| 领先指标 | 顾客平均取号耗时 | ≤ 15s | 45s | 上线后 7 天 | 埋点数据 |
| 滞后指标 | 客流流失率 | 降低 20% | 35% | 上线后 30 天 | 门店统计 |
| 滞后指标 | 顾客满意度 | ≥ 4.2 分 | 3.8 分 | 上线后 30 天 | 评价数据 |

## 八、持续优化
